# PollsVote 테스트 코드 안내

## 1. 사용된 테스트 기법

- **예외 및 경계값 테스트 (Exception & Boundary Testing)**: 정상적인 흐름(Happy Path)보다, 중복 투표나 잘못된 옵션 선택 등 시스템이 **실패해야 하는 상황(Sad Path)**에 집중하여 에러가 의도대로 던져지는지(`.rejects.toThrow`) 검증합니다.

## 2. 유사하거나 다른 기법 비교

- **해피 패스 테스트 (Happy Path Testing)**: 입력값이 완벽할 때 성공적으로 DB에 저장되고 200 OK를 반환하는지 위주로 검사합니다.
- 상태 기반 테스트(State-based Testing): 에러 발생 여부가 아니라 투표 카운트가 실제로 `+1` 되었는지를 검증하는 방식입니다.

## 3. 이 기법을 도입한 이유

투표를 행사하는 로직(Vote Casting)은 데이터 무결성이 매우 중요합니다. (중복 투표 방지, 마감된 투표 참여 방지 등)
따라서 비정상적인 접근이 들어왔을 때 트랜잭션이 오염되지 않고 즉시 적절한 `Error`를 방출하며 튕겨내는지를 확인하는 것이
투표 도메인에서 가장 치명적인 버그를 막는 방법이기 때문입니다.

# PollScheduler 테스트 코드 안내

## 1. 사용된 테스트 기법

- **스파이 및 테스트 더블 (Spies & Test Doubles)**: Express의 Request, Response, Next 객체를 `jest.fn()`을 이용해 가짜 객체로 만들어 컨트롤러 함수에 주입했습니다.

## 2. 유사하거나 다른 기법 비교

- **E2E (End-to-End) 테스트**: `Supertest` 라이브러리를 사용해 가상의 Express 앱을 띄우고 HTTP 요청을 직접 보내서 응답을 검증하는 방식입니다.
- 컨트롤러 단위 테스트(현재 기법)는 프레임워크 런타임 없이 함수 호출만으로 검증하므로 훨씬 가볍습니다.

## 3. 이 기법을 도입한 이유

현재 `pollScheduler`는 오직 개발용으로 서버의 헬스체크 및 쿼리 동작을 검증하기 위한 엔드포인트입니다.
무거운 Express 애플리케이션 생명주기를 테스트에서 재현할 필요 없이,
Controller가 정상적으로 Repository를 호출하는지, 그리고 성공/실패 시 `res.status`나 `next()`를 올바르게 호출하는지(제어 흐름)만 검증하기 위해 이 방식을 채택했습니다.

# Polls 테스트 코드 안내

## 1. 사용된 테스트 기법

- **데이터 픽스처 (Data Fixtures) & 스터빙 (Stubbing)**: 테스트에 필요한 고정된 가짜 데이터(`dummyPoll`)를 만들어두고, 의존하는 컴포넌트가 무조건 이 데이터를 반환하도록 스터빙(강제 지정)했습니다.

## 2. 유사하거나 다른 기법 비교

- **팩토리 패턴 (Factory Pattern)**: `Faker.js`나 `Rosie` 같은 라이브러리를 사용하여 테스트 실행 시마다 랜덤한 속성을 가진 객체를 동적으로 생성하는 기법입니다.
- 현재의 Fixture 방식은 정적이고 예측 가능한 데이터를 제공합니다.

## 3. 이 기법을 도입한 이유

투표(Poll) 데이터는 구조가 복잡(여러 옵션, 투표율, 상태 등)한 경우가 많습니다.
테스트마다 복잡한 객체를 직접 타이핑하는 것은 유지보수에 좋지 않으므로, 고정된 Fixture를 상단에 선언하여 재사용성을 높였습니다.
이를 통해 투표 객체의 비즈니스 로직(예: 마감된 투표인지 확인하는 로직 등)을 테스트할 때 일관된 환경을 보장받을 수 있습니다.

# Notifications 테스트 코드 안내

## 1. 사용된 테스트 기법

- **비동기 통신 모킹 (Asynchronous / Promise Mocking)**: `mockResolvedValue` 와 `mockRejectedValue`를 활용하여 외부 네트워크 통신(FCM, Email 발송 등)이 발생했을 때의 비동기 지연 및 응답을 시뮬레이션합니다.

## 2. 유사하거나 다른 기법 비교

- **Nock 또는 MSW (Mock Service Worker)**: `jest.mock`으로 함수 자체를 덮어씌우는 대신, 실제 HTTP 요청이 나갈 때 네트워크 레이어에서 가로채서 가짜 응답을 주는 방식입니다. (외부 API 연동 테스트에 더 정교함)

## 3. 이 기법을 도입한 이유

알림 전송 기능은 Firebase Cloud Messaging(FCM), 메일 서버(SMTP) 등 "외부 의존성"을 강하게 갖습니다.
테스트를 실행할 때마다 실제 사용자에게 푸시 알림이 가거나, 외부 API의 Rate Limit에 걸리는 상황을 방지해야 합니다.
따라서 외부 클라이언트를 직접 모킹하여, 네트워크 지연이나 장애(Timeout)가 발생했을 때
우리 서버가 뻗지 않고 유연하게 대처(에러 로깅 후 정상 응답 등)하는지 검증하기 위해 이 기법을 적용했습니다.

# Events 테스트 코드 안내

## 1. 사용된 테스트 기법

- **모듈 모킹 (Module Mocking)**: `jest.mock('./event.repository')`를 사용하여 데이터베이스(Prisma)에 직접 접근하는 Repository 계층을 가짜(Mock) 객체로 대체했습니다.

## 2. 유사하거나 다른 기법 비교

- **통합 테스트 (Integration Testing, ex: Supertest)**: 모킹 없이 실제 DB(또는 테스트용 인메모리 DB)까지 연결하여 Controller부터 Repository까지 통째로 검증하는 방식입니다. DB I/O가 발생하여 속도가 느립니다.
- **스텁 (Stubbing)**: 특정 함수의 반환값만 하드코딩해서 고정시키는 기법입니다. Mocking이 호출 횟수나 인자까지 추적(`toHaveBeenCalled`)한다면, Stub은 상태 반환에 집중합니다.

## 3. 이 기법을 도입한 이유

Service 계층의 주된 책임은 "비즈니스 로직(예: 날짜 범위 검증, 응답 객체 매핑)"을 수행하는 것입니다.
실제 데이터베이스가 없어도 로직의 참/거짓을 판별할 수 있어야 하므로, 데이터베이스 통신 책임을 가진 Repository를 모킹하여 **순수 단위 테스트(Unit Test)** 환경을 구축했습니다.
이를 통해 테스트 속도를 크게 높이고, 환경(DB 상태)에 의존하지 않는 안정적인 테스트가 가능해집니다.

# Notices 테스트 코드 안내

## 1. 사용된 테스트 기법

- **생명주기 훅 활용 (Setup & Teardown Hooks)**: `beforeEach`, `afterAll` 등의 Jest 훅을 사용하여 각 테스트 간의 **상태 격리(State Isolation)**를 보장합니다.

## 2. 유사하거나 다른 기법 비교

- **인라인 모킹 (Inline Mocking)**: 훅을 사용하지 않고 `it()` 블록 내부에서 매번 `jest.clearAllMocks()`나 리셋을 직접 호출하는 방식입니다. 테스트가 많아지면 중복 코드가 증가합니다.

## 3. 이 기법을 도입한 이유

Service 계층의 테스트(공지사항 생성, 조회, 수정, 삭제)가 많아지면 앞선 테스트에서의 모킹 호출 기록(`toHaveBeenCalledTimes`)이나 상태가 다음 테스트에 영향을 미쳐(Test Leak) 원인 모를 실패를 유발할 수 있습니다.
공지사항처럼 CRUD 로직이 단순하면서 반복적인 도메인은
`beforeEach`에서 모킹을 깔끔하게 리셋하고 공통 데이터를 세팅함으로써, 테스트 코드의 중복을 줄이고 신뢰성을 확보하는 것이 유리합니다.
